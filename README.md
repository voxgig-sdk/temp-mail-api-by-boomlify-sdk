## About Temp Mail API by Boomlify

[Boomlify](https://boomlify.com) provides a temporary (disposable) email service with a free public API. The service is aimed at developers and users who need throwaway inboxes for testing, signup flows, or avoiding spam — without the short lifetimes that other temp-mail services impose.

What the API lets you do:

- Create temporary email addresses on demand, optionally on a custom domain.
- Read messages delivered to those addresses via an inbox view.
- Manage the lifecycle of generated addresses (extended expiry options ranging from minutes to months).

Operational notes: requests are authenticated with a per-account API key obtained by registering at boomlify.com. The free tier includes a daily quota of credits (one credit per email created). The catalogue page for this API lives at [freepublicapis.com](https://freepublicapis.com/temp-mail-api-by-boomlify); the base URL for the API is `https://boomlify.com/api/v1`.

## Entities

The API exposes 3 entities:

| Entity | Description | API path |
| --- | --- | --- |
| **Domain** | Custom or shared domains that temporary email addresses can be issued under. | `/domains` |
| **Email** | An individual temporary email address generated by the API, with a configurable expiry. | `/email/create` |
| **Inbox** | The collection of messages delivered to a given temporary email address. | `/inbox/{email}` |

Each entity supports the following operations where available: **load**,
**list**, **create**, **update**, and **remove**.

## How it works

Every SDK call runs the same five-stage pipeline:

1. **Point** — resolve the API endpoint from the operation definition.
2. **Spec** — build the HTTP specification (URL, method, headers, body).
3. **Request** — send the HTTP request.
4. **Response** — receive and parse the response.
5. **Result** — extract the result data for the caller.

A feature hook fires at each stage (e.g. `PrePoint`, `PreSpec`,
`PreRequest`), so features can inspect or modify the pipeline without
forking the SDK.

### Features

| Feature | Purpose |
| --- | --- |
| **TestFeature** | In-memory mock transport for testing without a live server |

Pass custom features via the `extend` option at construction time.

### Direct and Prepare

For endpoints the entity model doesn't cover, use the low-level methods:

- **`direct(fetchargs)`** — build and send an HTTP request in one step.
- **`prepare(fetchargs)`** — build the request without sending it.

Both accept a map with `path`, `method`, `params`, `query`,
`headers`, and `body`. See the [How-to guides](#how-to-guides) below.
